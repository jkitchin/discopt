# C-40 fix — default-path false-optimal on `util` (MBQCP) (2026-07-07)

**Verdict:** a **class-(b) unsound presolve/bound-tightening**. The per-incumbent
**cutoff-FBBT** block mapped a Rust FBBT result whose variable layout (**144**
intervals) did **not** match the flat B&B column count (**145**); the block→flat
index map wrote a **crossed** `lb > ub` global box *in place* and swallowed the
resulting index-out-of-bounds, leaving the global `lb`/`ub` corrupted. The
corrupted global box then emptied the intersection at the child-node cutoff clamp
(`solver.py:5527`), fathoming the child that contains the true optimum. The
surviving frontier bound collapsed to the incumbent and the tree certified a
false-optimal `bound = 1072.96 > 999.58`. This is the *same failure family* as C-38
(a false-infeasible fathom of the optimum-containing node) but a **different
mechanism**: C-38's McCormick-LP Farkas fix is already present, and `util` still
failed after it.

Branch `fix-c40-util` from `origin/main` (`ba0668db`). Python-only change
(`python/discopt/solver.py`, one hunk); no Rust touched. Run env
`PYTHONPATH=<worktree>/python JAX_PLATFORMS=cpu JAX_ENABLE_X64=1`.

## Reproduce (confirmed on this build, all flags OFF/default)

```
from_nl('util.nl').solve(time_limit=60, gap_tolerance=1e-4)
# main:  status=optimal  obj=1072.961447893556  bound=1072.961447893556  nodes=3   <- FALSE
```
MINLPLib oracle: `=opt= 999.5787502`. `util` is **minimize** (linear objective;
four bilinear product constraints `v3*v0 + v2*v1`, `v3*v4`, `v3*v5`, `v3*v6`;
145 vars, 28 binary). For a minimize problem the certified dual (lower) bound
1072.96 exceeds a feasible objective 999.58 — an impossible underestimator, a
genuine false-optimal certificate. **Invariant violated:** the dual bound crossed a
feasible point (`bound ≤ true optimum` fails for min sense).

## Per-node trace — WHERE the bound crossed 999.58

The oracle optimum (`minlplib/sol/util.p1.sol`, obj 999.5787502) maps cleanly into
discopt's flat variable order via `minlplib/nl/util.col`: it lies inside the root
box, all bounds satisfied, objective evaluates to 999.5787502 exactly.

| stage | result on the optimum-containing box |
|---|---|
| root McCormick LP bound (full box, 7 nonlinear vars have `ub=+inf`) | no usable bound (∞ bounds) |
| Rust root FBBT (`tighten_root_bounds_with_fbbt`) | never excludes the optimum |
| Python nonlinear tightening (`_apply_nonlinear_tightening_with_status`) | never excludes the optimum |
| root OBBT (`obbt_tighten_root`, 127+ bounds) | never excludes the optimum |
| root McCormick LP on the OBBT-tightened box | **999.26** — a *valid* lower bound (≤ 999.58) |
| feasibility-pump / SubNLP incumbent | obj **1072.96** (a local optimum, not the global) |
| per-incumbent cutoff-OBBT/DBBT (cutoff 1072.96) | tightens 78 bounds; oracle **survives** |
| per-incumbent **cutoff-FBBT** (`_model_repr.fbbt_with_cutoff`) | **corrupts the global box**: `lb>ub` on ~17 vars ← bug site |
| child-node cutoff clamp (`solver.py:5527`) | **empty intersection** on the oracle child → fathomed as infeasible |

Tree-level `import_results` trace:

```
node 0 (root):   lb=943.66            -> branches, 2 children open
nodes 1 & 2:     lb=SENTINEL (both)   -> fathomed infeasible; open=0
                 global_lower_bound jumps to 1072.96 (the incumbent) -> "optimal"
```

The batch NLP (`_solve_batch_pounce`) returns child node 1 (which *contains* the
oracle) with a valid finite objective **999.55**; that valid bound is then discarded
because the node is sentineled. The sentinel fires at `solver.py:5527`:

```python
bl = np.maximum(batch_lb[i], _gl)   # _gl = global lb
bu = np.minimum(batch_ub[i], _gu)   # _gu = global ub
if np.any(bl > bu + 1e-9):
    node_infeasible_mask[i] = True  # <- fathom on an EMPTY intersection
```

Dumping the empty-intersection variables shows the **global box itself is crossed**,
e.g. `global var 68 = [104.22, 2.2]` (lb > ub), while the oracle has `x[68]=1.2`
inside the child box. The oracle lies inside the child box but outside the corrupted
global bound, so the intersection is empty and the node is fathomed.

## Root cause — class (b), unsound presolve/bound-tightening

The corrupted global box is produced by the per-incumbent **cutoff-FBBT** block
(`solver.py`, the `_model_repr.fbbt_with_cutoff` phase). During the solve,
`fbbt_with_cutoff` returns **144** intervals while the flat model has **145**
columns (a reformulated/eliminated `_model_repr` layout — confirmed: `len==144`,
`n_vars==145`). The old block→flat map,

```python
for bi, vinfo in enumerate(model._variables):
    if vinfo.size != 1:
        flat_idx += vinfo.size; continue
    for j in range(vinfo.size):
        if fbbt_lbs[bi] > lb[flat_idx] + 1e-10: lb[flat_idx] = fbbt_lbs[bi]   # <- misaligned
        if fbbt_ubs[bi] < ub[flat_idx] - 1e-10: ub[flat_idx] = fbbt_ubs[bi]
        flat_idx += 1
```

reads `fbbt_lbs[bi]` (block index into a 144-length array) and writes to
`lb[flat_idx]` (145-column flat index). With the lengths diverged, `bi` reads a
*misaligned* variable's bound; the write puts variable A's tightened bound onto
variable B, producing `lb[flat] > ub[flat]`, and at `bi=144` throws
`IndexError: index 144 out of bounds for axis 0 with size 144`. The write is done
**in place** on the shared global `lb`/`ub`, and the `IndexError` is swallowed by
the surrounding `except Exception`, so ~17 already-corrupted crossed entries persist
in the global box. (`fbbt_with_cutoff` itself is sound: on the fresh model it
returns 145 aligned, crossing-free intervals that keep the oracle.)

Line-trace confirmation: the crossed-bound count climbs 1→17 at the loop's write
line, then the swallowed OOB leaves the global box corrupted.

## The fix (general — the class, not the instance)

`python/discopt/solver.py`, one hunk in the cutoff-FBBT block; **no name/shape
special-casing**:

1. Apply the tightening only when the repr layout **provably aligns 1:1** with the
   flat columns — `fbbt_lbs.shape == (n_vars,)` **and** every model block scalar
   (block index == flat column). Otherwise skip and log; a looser box is always a
   valid bound (CLAUDE.md §3 — forgo an *optional* tightening, never a valid bound).
2. Build candidate bounds without in-place mutation (`np.maximum` / `np.minimum`)
   and **commit only a crossing-free intersection** (`not np.any(cand_lb > cand_ub)`).
   A crossing would mean the region is empty under the cutoff — a fact the tree
   proves via node relaxations, never via an in-place box write.

Sound by construction: the corrupted-box vector is eliminated, and the child-node
cutoff clamp can no longer see a crossed global box.

## Follow-up (tracked)

The 144-vs-145 length mismatch is a latent `model_to_repr` / `_model_repr` layout
divergence: the Rust FBBT repr drops/relayouts a variable relative to
`model._variables`. The fix soundly guards around it (skip on misalignment), so the
cutoff-FBBT tightening is currently *forgone* on any model with this divergence.
Restoring the tightening by exposing the repr's flat-column mapping (so
`fbbt_with_cutoff` results can be aligned exactly) is a performance follow-up, not a
soundness item — the current behavior is correct, just weaker on those models.
